# 终局视界：Agent-Native CLI vs MCP 量化实证研究

> 以高德开放平台 POI 服务为基准，结合飞书/钉钉 CLI 发布趋势

**版本: v3 严谨修正版** | 实验时间: 2026-04-03T00:21 CST

---

## 一、实验设计 (Methodology)

### 1.1 实验环境

| 项目 | 配置 |
| :--- | :--- |
| 运行时 | macOS, Python 3.9 (venv 隔离) |
| 大模型 | `MiniMax/MiniMax-M2.5` via ModelScope API |
| Token 计量 | `tiktoken cl100k_base` — 行业标准分词器 |
| 延迟采样 | 每组 N=3 次取均值，消除网络抖动 |
| 被测服务 | 高德开放平台 v3/place/text (POI 关键字搜索) |

### 1.2 对照组设计

| | Agent-Native CLI (实验组) | MCP Server (对照组) |
| :--- | :--- | :--- |
| **架构原型** | 类似 `feishu-cli` / `dingtalk-workspace-cli`，将高德 API 封装为 `amap-cli` | 标准 MCP JSON-RPC Server，暴露 `tools/list` |
| **能力发现** | 渐进式：`--help` → `search --help` | 全量：初始化时下发所有 Tool Schema |
| **输出协议** | stdout 纯 JSON，兼容 Unix 管道 (`jq`, `grep`) | JSON-RPC 结构化 tool role 返回 |
| **状态模型** | 无状态子进程 (fork → exec → exit) | 有状态长连接 (stdio/SSE) |

### 1.3 公平性原则

> [!IMPORTANT]
> v3 版本修正了前两版的关键方法学缺陷：
> 1. **CLI 侧同样计入 LLM 决策轮次**：Agent 需要先调用 LLM 生成 bash 命令，不能假设为零开销
> 2. **错误恢复测试对齐**：CLI 侧使用工具运行时错误（非 argparse 输入校验），与 MCP 侧对等
> 3. **上下文污染去除不对称**：CLI 不额外注入 system prompt，与 MCP 完全等量消息

## 二、量化数据 (Results)

### 实验 1：协议冷启动税 (Protocol Overhead)

Agent 在初始化时需要"了解"有哪些工具可用。对比两种架构的信息加载代价：

| 架构 | 加载方式 | 3 接口 | 10 接口 | 20 接口 |
| :--- | :--- | ---: | ---: | ---: |
| **CLI** | 按需 `--help` (仅 search) | **214 tokens** | **214 tokens** | **214 tokens** |
| **MCP** | 一次性全量 Schema | 482 tokens | 1,225 tokens | 2,280 tokens |
| **差距倍率** | | 2.3x | **5.7x** | **10.7x** |

```
Token 增长曲线:
CLI:  ████████████ (214)                    ← 恒定
MCP:  ██████████████████████ (482)          ← 3 tools
MCP:  ██████████████████████████████████████████████████████████ (1225)  ← 10 tools
MCP:  ██████████████████████████████████████████████████████████████████████████████████████████████████████████████████ (2280)  ← 20 tools
```

**关键发现**: CLI 的 Token 成本恒定（仅加载所需子命令的帮助），而 MCP 随接口数**线性增长**。飞书 CLI 暴露 200+ 命令，若走 MCP 全量下发将吞噬 ~20,000+ tokens。

### 实验 2：管道可组合性 (Composability) — 公平对照

任务指令：*"查一下北京的星巴克，只告诉我第一家的名字"*
两侧均从**同一 LLM 决策起点**出发。

| 指标 | CLI (bash + jq 管道) | MCP (Function Calling) |
| :--- | ---: | ---: |
| LLM 决策轮次 | 2 轮 (生成命令 + 总结提取) | 2 轮 (决策调用 + 消化结果) |
| LLM Token 消耗 (均值) | **263 tokens** | **1,357 tokens** |
| 端到端延迟 (均值) | **5.50s** | **6.40s** |
| 加速比 | **1.16x** | baseline |

**关键发现**:
- **Token 节省依然是降维打击**: 即便给予充分的对称性（要求 CLI 架构在管道执行后也经过一轮 LLM 进行总结），CLI 仅仅用了 **263 token** (生成一行 bash 命令 + 阅读十几字节的最终过滤结果) 便完成了任务。作为对比，MCP 强迫大模型在第二轮阅读几千字节的全量 JSON，总计挥霍 **1,357 token**，差额高达 **80%**。
- **延迟优势温和**: 端到端时间 CLI 快了 16%，但这与 Amap 网络 I/O 都是共同开销有关。真正的差异在第二轮：CLI 让 `jq` (C 程序) 做 JSON 提取近乎零耗，MCP 则强迫 LLM 用数百 Token 重新理解全量 JSON 再输出结果。

### 实验 3：错误恢复容灾 (Error Recovery)

任务："查询上海的苹果直营店"，首次调用返回 `INVALID_USER_KEY` 错误。

| 指标 | CLI | MCP |
| :--- | ---: | ---: |
| 错误检测耗时 | **0.106s** (exit code + stderr) | ~3s (需等 LLM 推理理解错误) |
| 恢复重试总用时 | **2.455s** | **3.978s** |
| 恢复消耗 Token | 0 (Agent 本地决策重试) | **1,305 tokens** |
| 状态污染 | 无 (无状态子进程) | 累积历史上下文 |

**关键发现**: CLI 的错误是操作系统级的信号（exit code 2 + JSON 错误体），Agent 无需消耗 Token 即可感知并重试。MCP 的错误必须被 LLM "阅读理解" 才能决定下一步，每次重试都复制一遍越来越大的完整历史。

### 实验 4：渐进式发现 vs 全量加载 (Discovery Pattern)

与实验 1 数据一致但视角不同——这里突出**现实场景**：Agent 本轮只需要搜索 POI，不需要路线规划和静态地图。

| 场景 | CLI 策略 | Token 代价 | MCP 策略 | Token 代价 |
| :--- | :--- | ---: | :--- | ---: |
| 只搜 POI | `search --help` | **214** | 加载全部 3 工具 | 482 |
| 只搜 POI | `search --help` | **214** | 加载全部 10 工具 | 1,225 |
| 只搜 POI | `search --help` | **214** | 加载全部 20 工具 | 2,280 |

**关键发现**: MCP 不支持 "只加载我需要的那个工具"——协议握手必须暴露所有 capabilities。CLI 天然按需。

### 实验 5：多轮上下文膨胀 (Context Pollution)

模拟 2 轮完整的搜索对话后，对比积累的上下文 Token 总量：

| | MCP | CLI | 差额 |
| :--- | ---: | ---: | ---: |
| 上下文总 Token | 295 | **271** | **CLI 少 24 tokens** |

逐条消息剖析（每轮一对）：

| 消息位 | 角色 | MCP Token | CLI Token | 差额 | 原因 |
| :---: | :--- | ---: | ---: | ---: | :--- |
| [1] | assistant (调用) | 47 | 24 | **-23** | MCP 含 tool_calls JSON 对象 |
| [2] | tool/user (结果) | 38 | 55 | +17 | CLI 含 "系统执行结果:" 包装文字 |
| [5] | assistant (调用) | 43 | 20 | **-23** | 同理 |
| [6] | tool/user (结果) | 30 | 47 | +17 | 同理 |

**关键发现**: 这是一个 **"此消彼长"** 的有趣平衡——
- MCP 的 `tool_calls` JSON 对象（含 id、type、function.arguments）本身很重，每次有 ~23 token 的结构化开销
- CLI 的执行结果嵌入 user 消息时有 "系统执行结果:\n...\n请根据结果回答。" 的包装文字增加了 ~17 token
- 总体 CLI 略胜，但差异不大 (8%)。**两者在多轮场景下几乎持平。**

## 三、综合评估矩阵

| 评估维度 | CLI 优势 | MCP 优势 | 量化差距 |
| :--- | :--- | :--- | :--- |
| 冷启动 Token 税 | ✅ 恒定 214 tokens | ❌ 线性增长 | **CLI 胜 2.3x~10.7x** |
| Token 效率 (单任务) | ✅ 263 tokens | ❌ 1,357 tokens | **CLI 节省 80%** |
| 端到端延迟 | ✅ 5.5s | ❌ 6.4s | **CLI 提速 16%** |
| 错误恢复速度 | ✅ 2.5s (0 token) | ❌ 4.0s (1,305 tokens) | **CLI 胜 38%** |
| 多轮上下文开销 | ≈ 持平 | ≈ 持平 | 差异 <10% |
| Unix 可组合性 | ✅ 原生管道 | ❌ 不支持 | **CLI 独占** |
| 工具动态发现 | ❌ 需 `--help` | ✅ 协议内建 | **MCP 胜** |
| 跨语言/跨端标准化 | ❌ 依赖 OS | ✅ 协议统一 | **MCP 胜** |

## 四、结论

### CLI 的绝对优势区
1. **Token 效率碾压 (节省 ~80%)**：管道组合过滤让大模型只需要咀嚼最终结果，规避了全量 JSON 污染上下文，是最大的实证发现。
2. **协议零税**：当平台接口众多时（如飞书 200+ API），CLI 的恒定 214 token 对比 MCP 的上万 token 全量注入是不可忽视的工程优势。
3. **无状态容灾**：进程崩溃不污染 Agent 运行时，可立即无损重试。

### MCP 仍然不可替代的场景
1. **跨网络架构 (Cloud-to-Local)**：这里的“远程”指的是 Agent 运行环境（如 Coze/Dify 云端、Web 浏览器应用、手机端）与工具数据所在环境**不在同一台物理机或容器内**。如果 Agent 在云端，它无法直接调用企业内网的 CLI；而 MCP 通过 SSE/长连接天然解决了跨网络的安全 RPC 调用。
2. **工具动态发现**：MCP 协议内建了 `tools/list` 和 `resources/list`，CLI 的 `--help` 缺乏跨语言一致的机器可读 Schema。
3. **复杂编排**：需要事务性、有状态的多步编排（如 OAuth 授权流）时，MCP 的会话管理更可靠。

### 最终判断

> **"CLI 不是 MCP 的竞争者，而是 MCP 的降维补充 (Complementary Downgrade)"**

在飞书、钉钉相继发布 CLI 工具的趋势背后，核心逻辑是：**对于驻扎在终端的 AI Agent (Claude Code, Aider, Cursor 等)，CLI 才是它们的母语，而 MCP 是外语。** 当任务可以用管道一行解决时，用 MCP 就是 overengineering。但当你需要企业级的权限审计、跨端互通、服务网格，MCP 依然是正确答案。

> [!TIP]
> **给开放平台的建议**: 同时维护 CLI + MCP 双通道。CLI 面向本地极客 Agent，MCP 面向云端编排平台。就像 GitHub 同时提供 `gh` CLI 和 REST API 一样。
